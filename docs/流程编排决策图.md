# 流程编排决策图

本文专门说明当前项目的“流程编排”是如何决定的。

目标回答两个问题：

1. 当前这轮到底会选中哪一套流程步骤。
2. 选中的步骤如何变成自动运行里的节点执行链。

## 1. 一句话结论

当前项目的流程编排不是写死在某一个脚本里，而是按下面这条链动态决定：

```text
配置输入
  -> 能力校验与模式归一化
  -> 选择当前模式下的步骤集合
  -> 步骤集合转节点图
  -> 节点图绑定真实执行器
  -> 自动运行按节点顺序执行
```

其中：

- “选择哪套步骤”主要由 `data/step-definitions.js` 决定
- “当前配置是否合法、是否允许切到该模式”主要由 `shared/flow-capabilities.js` 决定
- “节点图怎么跑起来”主要由 `background.js`、`background/workflow-engine.js`、`background/auto-run-controller.js` 决定

## 2. 总体关系图

```text
sidepanel / agent-control / 导入配置
  -> SAVE_SETTING / IMPORT_SETTINGS
  -> validateModeSwitch(...)
  -> resolveSignupMethod(...)
  -> 写入 state
  -> getOpenAiModeStepDefinitions(...)
  -> getSteps(...) / getNodes(...) / getWorkflow(...)
  -> buildStepRegistry(...)
  -> getStepRegistryForState(...)
  -> startAutoRunLoop(...)
  -> runAutoSequenceFromNodeGraph(...)
  -> executeNode(nodeId)
  -> stepExecutorsByKey[executeKey]
  -> background/steps/*.js 或 content/*.js
```

## 3. 决策输入项

当前流程编排主要由这些状态字段驱动：

- `activeFlowId`
- `panelMode`
- `plusModeEnabled`
- `plusPaymentMethod`
- `signupMethod`
- `resolvedSignupMethod`
- `plusAccountAccessStrategy`
- `phoneSignupReloginAfterBindEmailEnabled`

可以把它们分成三类：

### 3.1 平台接入方式

- `panelMode`
  - 例如：`local-cpa-json`
  - `local-cpa-json-no-rt`
  - `cpa`
  - `sub2api`
  - `codex2api`

### 3.2 主流程形态

- `plusModeEnabled`
  - 决定是普通注册链还是 Plus 链
- `plusPaymentMethod`
  - 在 Plus 链里决定走 `paypal / gopay / gpc-helper`
- `signupMethod`
  - 决定走邮箱注册还是手机号注册
- `resolvedSignupMethod`
  - 自动运行开始后冻结的实际注册方式

### 3.3 尾部行为和补链策略

- `plusAccountAccessStrategy`
  - 决定尾部是 OAuth 确认链，还是 `sub2api / cpa session import`
- `phoneSignupReloginAfterBindEmailEnabled`
  - 决定手机号链是否在绑定邮箱后再补一段重新登录链

## 4. 决策树

### 4.1 第 1 层：先看 flow

当前项目已经预留了多 flow 架构，但主 flow 仍然是 `openai`。

```text
activeFlowId
  -> openai
     -> 继续走 openai 的步骤定义
  -> 其他 flow
     -> 当前未注册对应执行器，直接报错
```

### 4.2 第 2 层：先看 panelMode

在 `openai` flow 内，最优先分流的是 `panelMode`。

```text
panelMode
  -> local-cpa-json-no-rt
     -> 走 hosted checkout 前缀 + oauth-login + confirm-oauth + local-cpa-json-export
  -> 其他 panelMode
     -> 继续看 plusModeEnabled
```

这个分支的特殊性在于：

- 它不是普通的完整 OAuth 尾链
- 而是明确走“本地无 RT 导出”的短链

### 4.3 第 3 层：看是否启用 Plus

```text
plusModeEnabled
  -> false
     -> 普通注册链
  -> true
     -> Plus 链
```

#### 普通注册链

```text
普通注册链
  -> signupMethod=email
     -> NORMAL_STEP_DEFINITIONS
  -> signupMethod=phone
     -> 如果启用 bind-email 后重登
        -> NORMAL_PHONE_BOUND_EMAIL_RELOGIN_STEP_DEFINITIONS
     -> 否则
        -> NORMAL_PHONE_STEP_DEFINITIONS
```

#### Plus 链

Plus 链会继续按支付方式分支：

```text
Plus 链
  -> plusPaymentMethod=paypal
  -> plusPaymentMethod=gopay
  -> plusPaymentMethod=gpc-helper
```

### 4.4 第 4 层：Plus 链里的支付方式分流

#### PayPal

```text
plusPaymentMethod=paypal
  -> hosted checkout final step 场景
     -> 先走 hosted checkout 前缀
     -> 再决定尾部
  -> 非 hosted final step 场景
     -> 走标准 PayPal Plus 链
```

#### GoPay

```text
plusPaymentMethod=gopay
  -> 使用 GoPay 前缀步骤集
  -> 再决定邮箱/手机号尾部
```

#### GPC Helper

```text
plusPaymentMethod=gpc-helper
  -> 使用 GPC 前缀步骤集
  -> 再决定邮箱/手机号尾部
```

### 4.5 第 5 层：尾部到底走 OAuth 还是 Session Import

这一层只在“邮箱注册 + Plus 模式”里真正生效。

```text
plusAccountAccessStrategy
  -> oauth
     -> 走 OAuth 登录确认尾链
  -> sub2api_codex_session
     -> 尾部替换成 sub2api-session-import
  -> cpa_codex_session
     -> 尾部替换成 cpa-session-import
```

也就是说：

- 前缀步骤通常还是注册 / checkout / payment
- 尾部步骤才根据接入策略切换

### 4.6 第 6 层：手机号补链策略

如果是手机号注册，还会再看：

```text
signupMethod=phone
  -> phoneSignupReloginAfterBindEmailEnabled=true
     -> 绑定邮箱后，再补一段“重新登录邮箱 + 验证码 + OAuth 确认”
  -> false
     -> 走更短的手机号尾链
```

## 5. 步骤集合如何变成节点图

步骤集合选出来之后，并不会直接按数组执行，而是会先转成节点。

这一步在 `data/step-definitions.js` 中完成：

```text
step
  -> id              -> legacyStepId
  -> key             -> nodeId
  -> order           -> displayOrder
  -> command / key   -> executeKey
  -> sourceId
  -> driverId
  -> mailRuleId
  -> next
```

然后会进入默认连线逻辑：

```text
如果当前 step 自己定义了 next
  -> 使用显式 next
否则
  -> 自动连到下一个 nodeId
```

所以当前项目的节点图默认是：

- 以线性链为主
- 保留显式 `next` 的扩展能力

## 6. 运行时选择的是哪套执行器

选完节点图以后，执行层还会再按当前 state 选择一次“活动 step registry”。

对应逻辑：

```text
getStepRegistryForState(state)
  -> local-cpa-json-no-rt ? localCpaJsonNoRtStepRegistry
  -> 非 Plus ? normal / normal-phone / normal-phone-relogin
  -> Plus + paypal ? plusPayPal...
  -> Plus + gopay ? plusGoPay...
  -> Plus + gpc-helper ? plusGpc...
```

也就是说：

- 步骤定义选了一次“当前有哪些节点”
- 执行器层又按同样的状态选了一次“当前该用哪套 registry 执行”

这两层必须一致，项目当前就是这样保持一致性的。

## 7. 自动运行如何消费这个节点图

自动运行控制器本身不重新定义流程，它负责的是：

- 第几轮
- 第几次尝试
- 是否续跑
- 从哪个节点开始

真正的运行顺序大致如下：

```text
startAutoRunLoop(...)
  -> autoRunLoop(...)
  -> runAutoSequenceFromNodeGraph(startNodeId, context)
  -> executeNode(nodeId)
  -> activeStepRegistry.executeNode(nodeId, state)
  -> background/steps/*.js 中的真实执行器
```

所以自动运行只是“消费编排结果”，不是“生成编排规则”的地方。

## 8. 为什么改配置后流程会重排

当设置发生变化时，后台不会只改一个布尔值，而是会重新做模式校验：

```text
SAVE_SETTING
  -> validateModeSwitch(...)
  -> resolveSignupMethod(...)
  -> 如果 stepModeChanged
     -> 重新计算 nextNodeIds
     -> nodeStatuses 全部重置为 pending
     -> currentNodeId 清空
```

因此，一旦下面这些字段变化：

- `plusModeEnabled`
- `plusPaymentMethod`
- `signupMethod`
- `phoneSignupReloginAfterBindEmailEnabled`
- `panelMode`

当前流程编排就会跟着切换，而不是沿用旧链路继续跑。

## 10. 最终总结

当前项目的流程编排可以概括为：

```text
模式输入决定步骤集合
步骤集合决定节点图
节点图决定执行顺序
自动运行只负责消费节点图
```

如果你要找“编排规则真正定义在哪里”，优先看：

1. `data/step-definitions.js`
2. `shared/flow-capabilities.js`

如果你要找“为什么切了模式之后整条流程都变了”，优先看：

1. `background/message-router.js` 的 `SAVE_SETTING`
2. `background.js` 的 `getStepRegistryForState(state)`

如果你只想单独看 `plusPaymentMethod=paypal` 这条分支，可继续参考 [PayPal编排决策图](./PayPal编排决策图.md)。
